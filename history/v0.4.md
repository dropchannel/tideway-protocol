# Winch Protocol — v0.4

## What Changed from v0.3

| Aspect | v0.3 | v0.4 |
|--------|------|------|
| First-class abstraction | `ChannelProvider` | `Channel` (composed of two physical pipelines) |
| Bidirectionality | Implicit (two slots configured per endpoint) | Explicit (definitional property of a channel) |
| Multi-hop | Not supported | Supported via `Node` primitive |
| Transport protocol | Delete-on-read; no delivery confirmation | Propagation + ACK cascade; delete-on-read is the ACK signal |
| `ChannelProvider` interface | `write`, `read`, `exists`, `delete` | Adds `peek()` — non-consuming read |
| Originating endpoint behavior | Polls recv-slot for response | Polls send-slot for ACK; response is a separate application concern |

**Rationale:** The v0.3 model worked well for single-hop use cases but encoded no concept
of pipeline composition and provided no delivery confirmation beyond "the slot is now
empty." The propagation protocol and ACK cascade fall out naturally from the existing
`ChannelProvider` abstraction — delivery confirmation is achieved by making deletion
propagate backward through the pipeline, requiring no additional state, messages, or
mechanism.

---

## Overview

v0.4 introduces the Winch propagation protocol: a formal definition of how a blob moves
from an originating endpoint to a terminating endpoint through any number of intermediate
nodes, and how delivery confirmation propagates back. It also formalises the Channel and
Node concepts and adds `peek()` to the `ChannelProvider` interface.

---

## Conceptual Model

### Channel

A **Channel** is a bidirectional logical communication pipeline between exactly two
endpoints. Bidirectionality is definitional — a channel always has two directions.

A channel is composed of exactly two directed **physical pipelines**, one per direction:

```
Endpoint A → [physical pipeline A→B] → Endpoint B
Endpoint A ← [physical pipeline B→A] ← Endpoint B
```

Each physical pipeline is independent. The two pipelines in a channel may use different
providers, different numbers of hops, and different infrastructure entirely.

### Physical Pipeline

A **physical pipeline** is a directed sequence of one or more `ChannelProvider` hops
carrying opaque encrypted blobs from one endpoint to the other. A single-hop pipeline
is two endpoints sharing a `ChannelProvider` directly — the v0.3 model. A multi-hop
pipeline inserts one or more Nodes between the endpoints.

### Endpoint

An **Endpoint** is a process that originates or consumes messages. Endpoints are the
only participants in a channel that perform encryption and decryption. An endpoint
configures one physical pipeline per direction: a send pipeline (outbound) and a receive
pipeline (inbound).

Endpoints have no knowledge of how many hops their pipelines contain, what providers
intermediate nodes use, or what the other endpoint looks like. They interact exclusively
with the `ChannelProvider` interface.

### Node

A **Node** is a process that forwards blobs along a physical pipeline from one
`ChannelProvider` to the next. Nodes do not encrypt, decrypt, or inspect message
content — blobs are opaque bytes to a node. A node has exactly one inbound provider
(recv-side) and one outbound provider (send-side). It belongs to exactly one physical
pipeline and operates in one direction only.

A node's configuration is entirely independent of the endpoints it serves. It has no
`SHARED_SECRET` and no knowledge of channel semantics.

### Separation of Concerns

| Concern | Owner |
|---------|-------|
| Encryption / decryption | Endpoint only |
| Message semantics (query/response, fire-and-forget, etc.) | Client application layer |
| Blob transport | ChannelProvider |
| Multi-hop composition | Node configuration |
| Delivery confirmation | Propagation protocol (ACK cascade) |
| Backpressure (one message per slot at a time) | ACK cascade (send-slot occupied = in-flight) |

The channel infrastructure has no opinion on message semantics. Whether Endpoint A always
initiates and Endpoint B always responds, or both send independently, or messages are
fire-and-forget — all of this is determined by the client application, not the channel.

## `peek()` Addition to ChannelProvider

`peek()` is added as a fifth operation on `ChannelProvider`. It reads a blob from a slot
without consuming it — the blob remains in the slot after the call.

`peek()` is required by the propagation protocol: during the forward pass, each
intermediate participant must be able to read the blob to forward it without triggering
the ACK cascade prematurely. Using `read()` (consuming) would delete the recv-slot,
breaking the backward cascade.

Full interface specification: [`spec/channel-provider.md`](https://github.com/dropchannel/spec/blob/main/channel-provider.md).

## Propagation Protocol

The propagation protocol defines how a blob moves from the originating endpoint to the
terminating endpoint through any number of intermediate nodes, and how delivery
confirmation propagates back to the originator.

### Forward Pass: Payload Propagation

Each participant in the pipeline — originating endpoint and every intermediate node —
reads its recv-slot using `peek()` (non-consuming) and writes the blob forward to its
send-slot. No participant deletes its recv-slot during the forward pass. The blob
accumulates at every storage point along the pipeline simultaneously.

```
After full forward propagation (two-hop example):

  Endpoint A         Node              Endpoint B
  [send: blob] →  [recv: blob]  →   [recv: blob]
                  [send: blob]
```

Every slot along the pipeline is occupied. The originating endpoint's send-slot,
every node's recv-slot and send-slot, and the terminating endpoint's recv-slot all
contain the blob.

### Backward Pass: ACK Cascade

1. **Terminating endpoint** reads its recv-slot with `read()` (consuming), which
   deletes it. This is the only `read()` call on any recv-slot in the entire protocol.
   The terminating endpoint decrypts and consumes the message.

2. **Final node** polls its send-slot with `exists()`. The terminating endpoint's
   `read()` deleted that slot. The node detects `exists()` → false — the ACK signal.
   The node calls `delete()` on its own recv-slot.

3. **Each preceding node** likewise polls its send-slot, detects the deletion caused
   by the downstream node's `delete()`, and deletes its own recv-slot.

4. **Originating endpoint** polls its send-slot. The first node's `delete()` cleared
   it. The originating endpoint detects `exists()` → false: delivery is confirmed. The
   endpoint is now free to send the next message.

```
After full ACK cascade (two-hop example):

  Endpoint A         Node              Endpoint B
  [send: empty] ← [recv: empty] ←  [recv: deleted by B]
                  [send: empty]
```

All slots empty. The pipeline is ready for the next message.

### Key Properties

**Crash safety.** If any participant crashes mid-propagation, the blob remains durably
in every slot already written. On restart, each participant reconstructs its exact state
by inspecting its two slots. No external coordination or log is required.

**Built-in backpressure.** The originating endpoint's send-slot remains occupied for the
entire round-trip duration. The endpoint cannot send a second message until the ACK
cascade completes. This is inherent to the protocol — no additional throttling mechanism
is needed.

**Delivery confirmation.** The originating endpoint receives implicit confirmation that
the terminating endpoint consumed the message. The confirmation signal is the deletion of
its send-slot, caused deterministically by the ACK cascade.

**No in-memory buffering.** No participant holds a blob in memory between poll cycles.
Blobs are always durably stored at every hop. A node that has peeked but not yet written
simply retries on the next poll cycle.

## Node Lifecycle

A node's behavior is fully determined by the observable state of its two slots. The node
has a startup inspection phase and two steady-state polling modes.

### Startup: Inspect Both Slots

On startup, the node calls `peek()` on both its recv-slot and send-slot once to determine
its starting state, then transitions into the appropriate steady-state mode.

| Recv-slot | Send-slot | Condition | Action |
|-----------|-----------|-----------|--------|
| Empty | Empty | Idle | → Recv-polling mode |
| Occupied | Empty | Unforwarded payload | `peek()` recv → `write()` send → Send-polling mode |
| Occupied | Occupied (match) | Already forwarded; awaiting ACK | → Send-polling mode |
| Occupied | Occupied (differ) | State corruption | Log blob sizes and SHA-256 hashes, halt |
| Empty | Occupied | Impossible state | Log, halt |

**Payload comparison** is raw byte equality — no decryption needed. Matching payloads
mean the forward pass completed before a crash and the node simply awaits the downstream
ACK. Differing payloads indicate partial write or state corruption not automatically
recoverable in v0.4.

### Recv-Polling Mode (steady state: idle)

The node polls its recv-slot only. The send-slot is assumed empty.

```
Loop:
  exists(recv_slot)?
    No  → sleep(poll_interval), repeat
    Yes → blob = peek(recv_slot)
          write(send_slot, blob)
          → transition to Send-polling mode
```

### Send-Polling Mode (steady state: in-flight)

The node polls its send-slot only. The recv-slot is assumed occupied.

```
Loop:
  exists(send_slot)?
    Yes → sleep(poll_interval), repeat   # downstream not yet ACK'd
    No  → delete(recv_slot)              # ACK received; clear recv-slot
          → transition to Recv-polling mode
```

### Polling Cost

| Phase | Operations per cycle |
|-------|---------------------|
| Startup | Two `peek()` calls, once only |
| Recv-polling mode | One `exists()` |
| Recv→Send transition | One `peek()` + one `write()` |
| Send-polling mode | One `exists()` |
| Send→Recv transition (ACK) | One `delete()` |

## Originating Endpoint Behavior

Under the propagation protocol, the originating endpoint's `send` operation changes from
v0.3:

- **v0.3:** Write blob to send-slot, then poll recv-slot for a response blob.
- **v0.4:** Write blob to send-slot, then poll send-slot `exists()` for the ACK signal.
  When `exists()` returns false, delivery is confirmed.

Polling the recv-slot for a response is a client application concern, not a channel
protocol concern. The return pipeline is always available for the responding endpoint to
write into independently.

The terminating endpoint's `listen` behavior is unchanged: it calls `read()` (consuming)
on its recv-slot, which is the correct terminating-endpoint behavior.

## Node Configuration

A node is configured entirely via environment variables. It requires no `SHARED_SECRET`.
Each side (recv and send) selects its provider independently via `RECV_PROVIDER` and
`SEND_PROVIDER`, with fallback to `CHANNEL_PROVIDER`. Provider-specific env vars are
namespaced by direction to avoid collision when both sides use the same provider type.

Example (node bridging a local filesystem recv-side to an HTTP relay send-side):

```
CHANNEL_ID=my-channel
RECV_PROVIDER=local
RECV_LOCAL_CHANNEL_PATH=/data/dropchannel
RECV_SLOT=a-to-b
SEND_PROVIDER=httprelay
SEND_RELAY_URL=https://<relay-url>
SEND_SLOT=a-to-b
```

The slot name is the same on both sides — it does not change across hops.

## Multi-Hop Topology Example

```
Endpoint A  →  [Local]  →  Node  →  [HTTP
 relay]  →  Endpoint B
Endpoint A  ←  [HTTP relay]  ←  Node  ←  [Local]  ←  Endpoint B
```

For bidirectional relay, two nodes are deployed — one per physical pipeline, each
configured independently. Each node operates in one direction only.

## Out of Scope (Post-v0.4)

- Automatic recovery from Occupied/Occupied payload-mismatch error state
- More than one message in flight per slot
- Atomic write guards (compare-and-swap) for concurrent writers
- Node fan-out (one inbound, multiple outbound)
- Node health checks or automatic restart (addressed in v0.6 via heartbeat protocol)
- Package structure and crypto isolation enforcement (addressed in v0.5)
