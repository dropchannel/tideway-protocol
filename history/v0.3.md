# Winch Protocol — v0.3

## What Changed from v0.2

| Aspect | v0.2 | v0.3 |
|--------|------|------|
| Storage medium | GCS via HTTP relay (only option) | Pluggable: any `ChannelProvider` implementation |
| Infrastructure requirement | HTTP relay server required | Provider-dependent; Dropbox and local filesystem require no server |
| Protocol dependency on medium | Implicit (HTTP semantics throughout) | Eliminated — protocol operates against the `ChannelProvider` interface only |

**Rationale:** The core protocol — symmetric peers, encrypted blobs, delete-on-read,
write guard — is medium-agnostic. Binding it to GCS/HTTP was an initial implementation
choice, not a protocol requirement. Abstracting the storage medium behind a
`ChannelProvider` interface makes the protocol usable over any shared storage location.

---

## Overview

v0.3 introduces the `ChannelProvider` abstraction: a formal interface that decouples the
Winch protocol from any specific storage medium. The protocol's behavior — symmetric
peers, slot-based blob exchange, delete-on-read, write guard — is unchanged. What changes
is that the storage layer is now a pluggable dependency rather than a fixed choice.

The `ChannelProvider` interface is defined at the system level and shared across all
protocols. See [`spec/channel-provider.md`](https://github.com/dropchannel/spec/blob/main/channel-provider.md).

---

## Protocol Behavior

All protocol behavior from v0.2 is unchanged:

- Symmetric peers, each configured with a `SEND_SLOT` and `RECV_SLOT`
- Blob lifecycle: slot presence = pending, slot absence = handled
- Retrieval is destructive (delete-on-read)
- Write guard: deposit rejected if slot already occupied

What v0.3 adds is that all of these operations are now expressed in terms of the
`ChannelProvider` interface rather than any specific storage API.

---

## ChannelProvider Interface (v0.3)

In v0.3 the interface has four operations. (`peek()` is added in v0.4.)

| Operation | Behavior |
|-----------|----------|
| `write(channel_id, slot, data)` | Deposit blob. Returns `True` on success, `False` if slot occupied. |
| `read(channel_id, slot)` | Retrieve and delete blob. Returns blob bytes or `None` if empty. |
| `exists(channel_id, slot)` | Returns `True` if a blob is pending, `False` if slot empty. |
| `delete(channel_id, slot)` | Delete blob. Idempotent — no error if already absent. |

Full interface specification: [`spec/channel-provider.md`](https://github.com/dropchannel/spec/blob/main/channel-provider.md).

---

## Write Guard Atomicity

The write guard is enforced by an `exists()` check before `write()`. On all current
providers this check-then-write sequence has an inherent race window — it is not
atomically guaranteed. At human interaction cadence and with single-writer usage patterns
this is acceptable. True atomic write guards (e.g. compare-and-swap, conditional PUT)
are deferred to a future revision.

---

## Provider Topology

The introduction of pluggable providers enables topologies that were not possible in
v0.1–v0.2:

**Direct filesystem (same machine):** Both peers on the same host share a local
directory. No network, no relay server. The channel functions as a backpressure
primitive between two local processes.

**Synced folder (cross-machine, no relay):** Both peers point at a shared synced
folder (e.g. Dropbox). The sync agent handles cross-machine transport transparently.
No relay server is deployed.

**HTTP relay (unchanged from v0.2):** A relay server mediates access to a remote
storage backend. Peers communicate with the relay via HTTP; the relay translates calls
to the underlying storage provider.

---

## Out of Scope (Post-v0.3)

- `peek()` — non-consuming read (addressed in v0.4)
- Multi-hop pipelines via Node primitive (addressed in v0.4)
- Delivery confirmation beyond delete-on-read (addressed in v0.4)
- Atomic write guards (compare-and-swap)
- More than two peers per channel
