# Winch Protocol — v0.2

## What Changed from v0.1

| Aspect | v0.1 | v0.2 |
|--------|------|------|
| Peer roles | Asymmetric: Sender and Receiver with distinct behaviors | Symmetric: both peers are identical, configured differently |
| Slot names | Hardcoded `request` / `response` | Arbitrary strings agreed out-of-band |
| Cross-slot deletion | Receiver's deposit triggers deletion of request slot | Removed — deletion is exclusively delete-on-read |
| Write guard | Not present | One blob per slot enforced; deposit rejected if slot occupied |

**Rationale:** Hardcoded slot names encoded topology into the relay, preventing reuse
across different peer configurations. Symmetric peers with generic slot names make the
relay pure infrastructure. All topology knowledge moves to peer configuration.

---

## Overview

v0.2 evolves the protocol from an asymmetric request/response model to a symmetric,
slot-based channel. Both peers are identical in capability; their roles in any given
exchange are determined entirely by which slots they are configured to read and write.
The relay has no knowledge of peer roles.

---

## Channel Model

A channel is identified by a **channel ID**. Within a channel, each peer is configured
with:

- **SEND_SLOT** — the slot name this peer writes to
- **RECV_SLOT** — the slot name this peer reads from

Slot names are arbitrary strings agreed out-of-band between the two peers. The relay
treats them as opaque identifiers. A correctly configured channel has the two peers'
slots mirrored: Peer A's `SEND_SLOT` is Peer B's `RECV_SLOT`, and vice versa.

Example:

```
Peer A:  SEND_SLOT=a-to-b   RECV_SLOT=b-to-a
Peer B:  SEND_SLOT=b-to-a   RECV_SLOT=a-to-b
```

---

## Peer Behavior

Both peers run identical logic. Each peer:

- **Deposits** encrypted blobs into its `SEND_SLOT`
- **Polls** its `RECV_SLOT` for incoming blobs
- **Decrypts** blobs on retrieval

There is no Sender or Receiver role at the protocol level. Either peer may initiate a
message at any time.

---

## Blob Lifecycle

- **Slot presence = pending. Slot absence = handled.**
- A blob deposited into a slot persists until the reading peer retrieves it.
- Retrieval is destructive: the blob is deleted as part of the read operation.
- Deletion is exclusively delete-on-read. There is no cross-slot coupling — depositing
  into one slot never causes deletion in another slot.
- The relay is stateless beyond the blobs it currently holds.

---

## Write Guard

A peer cannot deposit a blob into a slot that is already occupied. If a deposit is
attempted on an occupied slot, the relay rejects it. The depositing peer is responsible
for surfacing this condition.

This enforces the invariant that at most one unread blob exists per slot at any time.
No sequencing, queuing, or counters are required.

## Out of Scope (Post-v0.2)

- Multiple messages in flight per slot (addressed in v0.4 via the propagation protocol)
- Concurrent writers to the same slot
- More than two peers per channel
- Provider abstraction (addressed in v0.3)
