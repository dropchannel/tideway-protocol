# Winch Protocol — v0.6

## What Changed from v0.5

| Aspect | v0.5 | v0.6 |
|--------|------|------|
| Pipeline observability | None — stalls are opaque | Per-hop heartbeat chain visible to endpoints |
| Stall vs. offline distinction | Indistinguishable | Inferrable from heartbeat presence and `alive_s` |
| Intentional shutdown signal | None | Client `closing` status in heartbeat |
| Node slot model | recv-slot + send-slot | Adds meta recv-slot + meta send-slot |
| Participant identity | None | Persistent UUID per node and endpoint |
| Primary slot behavior | As specified in v0.1–v0.4 | **Unchanged** |

**Rationale:** A stalled pipeline is indistinguishable from a pipeline with an offline
participant, and there is no mechanism to identify which specific hop in a multi-hop
chain is responsible for a stall. Both gaps are resolved entirely within the existing
`ChannelProvider` interface via a meta slot layer. No new storage operations and no
out-of-band channels are required.

---

## Overview

v0.6 introduces the heartbeat protocol — a meta slot layer that runs independently of
the primary payload channel on every node and endpoint. Heartbeats provide continuous
pipeline observability without modifying any primary slot semantics.

---

## Extended Slot Model

Each participant gains two additional slots alongside its existing primary slots:

```
recv-side:  [ primary recv-slot ]  [ meta recv-slot ]
send-side:  [ primary send-slot ]  [ meta send-slot ]
```

Meta slots are logically separate storage namespaces within the same `ChannelProvider`
position. There is no coupling, no sequencing dependency, and no shared state between
the primary and meta layers. Primary slot operations are entirely unaffected by meta
slot state.

---

## Participant Identity

Each node and endpoint has a persistent UUID (UUID4) generated on first startup and
reloaded on all subsequent startups. The UUID is stable across restarts within a
deployment. Manually deleting the identity file causes a new UUID to be generated.

Identity is expressed in heartbeat content as `Node_<UUID>` for nodes and
`Client_<UUID>` for endpoints.

---

## Heartbeat File Naming

```
heartbeat-node-<UUID>      # written by a node
heartbeat-client-<UUID>    # written by a client (endpoint)
```

---

## Heartbeat Content Formats

**Node heartbeat:**

```
<upstream_content>
Node_<UUID>:<timestamp>:<alive_s>
```

**Client heartbeat:**

```
Client_<UUID>:<timestamp>:<alive_s>:<status>:<status_s>
```

| Field | Description |
|-------|-------------|
| `UUID` | Persistent UUID for this participant |
| `timestamp` | Unix timestamp (seconds) at time of write |
| `alive_s` | Seconds since this participant started its current session |
| `status` | Client only. One of: `ready`, `busy`, `closing` |
| `status_s` | Client only. Seconds since current status was set |

`upstream_content` is either the full contents of the one non-self heartbeat file found
on the relevant meta slot, or the literal string `HEARTBEAT_NOT_FOUND`. The writing
node's own line is always appended unconditionally.

---

## Node Heartbeat Cycle

Each node runs the following cycle at `HEARTBEAT_INTERVAL` (HI), concurrently with
and independently of the primary slot state machine.

**Step 1 — Read send-side meta, write recv-side meta:**

```
Look at send-side meta slot for any file heartbeat-node-* or heartbeat-client-*
where filename != heartbeat-node-<UUID_self>

If found:   content = read(send-side meta, <that file>)
If not:     content = "HEARTBEAT_NOT_FOUND"

write(recv-side meta, heartbeat-node-<UUID_self>,
      content + "\n" + "Node_" + UUID_self + ":" + timestamp + ":" + alive_s)
```

**Step 2 — Read recv-side meta, write send-side meta:**

```
Look at recv-side meta slot for any file heartbeat-node-* or heartbeat-client-*
where filename != heartbeat-node-<UUID_self>

If found:   content = read(recv-side meta, <that file>)
If not:     content = "HEARTBEAT_NOT_FOUND"

write(send-side meta, heartbeat-node-<UUID_self>,
      content + "\n" + "Node_" + UUID_self + ":" + timestamp + ":" + alive_s)
```

**Step 3 — Sleep HI seconds. Repeat from Step 1.**

If more than one non-self heartbeat file is found on a given side, the node selects the
most recently written and logs a warning.

---

## Client Heartbeat

Clients write only their own `heartbeat-client-<UUID>` to both meta slots. Clients do
not read or propagate any other heartbeat files.

Client heartbeats are event-driven at status transitions, then HI-driven until the next
transition. On each status transition, the client writes a heartbeat immediately and
resets its HI timer. Every HI interval thereafter, while the status remains unchanged,
the client writes another heartbeat with the same status and updated `timestamp`,
`alive_s`, and `status_s`.

| Trigger | Status written |
|---------|----------------|
| Startup | `ready` |
| Returning to idle (was busy, no message available) | `ready` |
| New message pickup (including busy → busy) | `busy` |
| Shutdown | `closing` |

`status_s` resets to 0 on every transition including `busy` → `busy`. During HI-driven
repeat writes, `status_s` increments normally — it reflects time in the current status,
not time since last write.

`closing` is written once on shutdown. No subsequent HI writes follow.

---

## Startup Purge

On startup, before beginning the heartbeat cycle:

- A node MUST delete all files matching `heartbeat-node-*` from both its recv-side and
  send-side meta slots.
- A client MUST delete all files matching `heartbeat-client-*` from both its recv-side
  and send-side meta slots.

Each participant purges only its own file type. Files written by other participants are
left intact. Recovery from the purge is bounded at 1×HI.

## Meta Slot Representation per Provider

| Provider | Meta slot representation |
|----------|--------------------------|
| `gcs` | Object prefix: `<channel_id>/meta/` |
| `dropbox` | Subdirectory: `<channel_path>/meta/` |
| `local` | Subdirectory: `<channel_dir>/meta/` |
| `httprelay` | URL namespace: `<channel_id>/meta/<filename>` |

## Reading Heartbeat State

A healthy two-hop pipeline (`ClientA → Node → ClientB`) produces a heartbeat chain
readable at ClientA's recv-side meta slot:

```
Client_<B-UUID>:<timestamp>:<alive_s>:ready:<status_s>
Node_<N-UUID>:<timestamp>:<alive_s>
```

The chain is assembled by the node relaying ClientB's heartbeat content with its own
line appended. ClientA can inspect this to determine:

- Whether ClientB is alive (its heartbeat file is present and `timestamp` is recent)
- Whether the node is alive (its line is present and `timestamp` is recent)
- Whether ClientB is idle, busy, or shutting down (`status` field)
- How long each participant has been in its current state (`alive_s`, `status_s`)

A broken hop produces `HEARTBEAT_NOT_FOUND` at the position of the offline participant,
allowing the specific stalled hop to be identified.

---

## Security

Heartbeat content is **unencrypted**. It consists solely of UUIDs, timestamps, integer
counters, status strings, and `HEARTBEAT_NOT_FOUND`. No payload content and no routing
secrets are present.

UUID exposure reveals only that a participant with that identifier exists and is active —
information already implicit in the existence of the channel configuration.

Encryption of heartbeat content via a dedicated key (`DRIP_KEY`) is deferred to a
future revision.

---

## Backward Compatibility

All primary slot behavior from v0.5 is unchanged. A v0.5 participant in a v0.6 pipeline
will not write heartbeat files and will appear as `HEARTBEAT_NOT_FOUND` at its adjacent
nodes. The pipeline continues to function; that hop is invisible to heartbeat diagnostics.

In a mixed v0.5/v0.6 pipeline, `HEARTBEAT_NOT_FOUND` cannot be used to distinguish an
offline participant from a non-participating v0.5 participant.

---

## Out of Scope (Post-v0.6)

- `DRIP_KEY` encryption of heartbeat content
- Heartbeat-based alerting or notification (left to the client application layer)
- Configurable `ABANDON_THRESHOLD` inference from heartbeat staleness
- Automatic recovery from stalled pipelines based on heartbeat state
