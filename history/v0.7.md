# Tideway Protocol — v0.7

**Status:** Adopted (ADR-037)
**Scope:** Vocabulary update — participant, address, and directional terminology aligned to waterway model. No behavioral change.

---

## Summary

v0.7 is a pure vocabulary update driven by ADR-037. The protocol state machine is
unchanged. No participant behavior, propagation logic, or payload semantics are altered.

The changes in this version are:

1. **Protocol renamed.** Winch → Tideway. The Waterway name prefix changes from
   `winch-` to `tideway-`. All Waterways named `winch-*` in existing deployments must
   be migrated.

2. **Forwarding participant renamed.** Node → Raft. This reflects the passive nature of
   the forwarding role — a Raft is carried by the tide, not an agent making independent
   decisions.

3. **Address model updated.** The implicit single-blob-per-Waterway model is retired. A
   payload is now addressed by `(channel, waterway, filename)`. The `channel_id`
   parameter is renamed to `channel` throughout. The Tideway-specific filename scheme
   (which encodes generation parity to signal direction) is deferred to a future version
   — see Open Items.

4. **Directional references updated.** `recv_backend` / `send_backend` are retired. A
   Raft now operates against two named Docks: `upper_dock` (the initiator-side Dock) and
   `lower_dock` (the responder-side Dock). For Tideway, even-generation payloads flow
   Upper → Lower; odd-generation payloads flow Lower → Upper. Upper and Lower are fixed
   — they do not swap regardless of which direction the current payload is traveling.

---

## What Changed from v0.6

| Aspect | v0.6 | v0.7 |
|--------|------|------|
| Protocol name | Winch | Tideway |
| Forwarding participant | Node | Raft |
| Storage interface | ChannelProvider | DockProvider |
| Interface address model | `(channel_id, waterway)` | `(channel, waterway, filename)` |
| Primary payload behavior | As specified in v0.1–v0.6 | **Unchanged** |

---

## DockProvider Address Model

The interface is renamed from `ChannelProvider` to `DockProvider` (full definition:
[`dropchannel/spec/channel-provider.md`](https://github.com/dropchannel/spec/blob/main/channel-provider.md)).
All operations now address files by three coordinates: `(channel, waterway, filename)`,
replacing the implicit single-blob-per-Waterway model. The `channel_id` parameter is
renamed to `channel` throughout.

---

## Backward Compatibility

v0.7's `(channel, waterway, filename)` address model is not wire-compatible with v0.6's
`(channel_id, waterway)` model. Mixed v0.6/v0.7 deployments are not supported.

---

## Open Items

**Raft file discovery without `list()`**
The `DockProvider` interface requires an explicit `filename` for all primary operations.
A `list(channel, waterway)` operation returning current filenames within a Waterway is
identified in ADR-037 as a future addition, deferred to a separate ADR. Until `list()`
is available, the Raft startup inspection and recv-polling state machine cannot enumerate
Waterway contents through the protocol interface alone.

**Startup role determination**
How a Raft determines direction on first startup (when no prior Waterway state has been
observed) is deferred to a future revision. This requires either `list()` or a
protocol-level convention for seeding the initial filename.
