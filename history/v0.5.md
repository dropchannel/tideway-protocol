# Winch Protocol — v0.5

## What Changed from v0.4

No protocol behavior changes in this version.

| Aspect | v0.4 | v0.5 |
|--------|------|------|
| HTTP relay provider identifier | `http` | `httprelay` |
| `CHANNEL_PROVIDER=http` | Valid | Replaced by `CHANNEL_PROVIDER=httprelay` |

**Rationale:** The identifier `http` implied that HTTP is the protocol's transport
mechanism, which is misleading. The HTTP relay is one *drop mechanism* — a shared
location where blobs are deposited and retrieved — that happens to be accessed via HTTP.
`httprelay` aligns with the other provider identifiers (`gcs`, `dropbox`, `local`), all
of which describe the drop mechanism rather than the transfer protocol.

The rename is a clean break. `CHANNEL_PROVIDER=http` is not supported as a legacy alias.
Existing configurations must update to `CHANNEL_PROVIDER=httprelay`.

---

## Implementation Notes

v0.5 also introduced a four-package monorepo split (`dropchannel-channel`,
`dropchannel-endpoint`, `dropchannel-node`, `dropchannel-server`) and enforced the
node's crypto isolation as a hard package-dependency property rather than a convention.
These are implementation concerns documented in `dropchannel-py`.
